01

What is JavaScript?

The Official Definition

JavaScript is a high-level, interpreted, single-threaded, dynamic, weakly-typed, prototype-based, multi-paradigm programming language with first-class functions.

Each of those words carries meaning. Let's decode them.

High-Level

You don't manage memory manually. No malloc(), no pointers. JavaScript abstracts hardware details. The engine handles memory allocation and garbage collection.

Interpreted

Code is NOT compiled to machine code ahead of time. The JS engine reads and executes code line by line at runtime. Modern engines use JIT (Just-In-Time) compilation for speed.

Dynamic

Variables don't have fixed types. A variable holding a number can later hold a string. Type is determined at runtime, not compile time.

Single-Threaded

JavaScript runs on one thread — one task at a time. Yet it handles async operations (API calls, timers) through the Event Loop. More on this in section 07.

Prototype-Based

Instead of classical class-based inheritance (like Java/C#), JS objects inherit directly from other objects via the prototype chain. ES6 class syntax is just sugar over this.

First-Class Functions

Functions are values. You can assign a function to a variable, pass it as an argument, return it from another function. This enables functional programming patterns.

Created in 10 days — Brendan Eich created JavaScript in 1995 for Netscape. Despite its rushed origin, it became the only language that runs natively in browsers — making it the most deployed language on Earth.

02

How JavaScript Runs — The Engine

The JS Engine

JavaScript needs an engine to execute. The most important one:

  • V8 — Built by Google. Used in Chrome and Node.js. Written in C++.
  • SpiderMonkey — Mozilla Firefox
  • JavaScriptCore (Nitro) — Safari / WebKit

What the V8 Engine Does

When V8 encounters your JS code, it goes through these phases:

// Source Code → Parsing → AST → Bytecode → Machine Code

1. Parsing     — Reads text, checks syntax, reports SyntaxErrors
2. AST         — Builds an Abstract Syntax Tree (tree of code structure)
3. Ignition    — V8's interpreter; converts AST → bytecode
4. TurboFan    — JIT compiler; "hot" (frequently-run) code compiled to machine code
5. Execution   — CPU runs the optimized machine code

Node.js = V8 engine + extra C++ APIs (filesystem, network, OS access). This is why JS can run on servers — V8 runs anywhere, and Node wraps it with system-level capabilities.

Data Types — The Building Blocks

Two Categories

PRIMITIVE (Value Types)

  • string — text
  • number — integers & floats (one type!)
  • bigint — very large integers
  • boolean — true / false
  • undefined — declared, not assigned
  • null — intentional absence
  • symbol — unique identifier (ES6)

REFERENCE (Object Types)

  • object — key-value pairs
  • array — ordered list (is an object!)
  • function — callable object
  • Date, RegExp, Map, Set — built-in objects

Primitive vs Reference — Critical Difference

// PRIMITIVES are copied by VALUE
let a = 10;
let b = a;          // b gets a COPY of 10
b = 20;
console.log(a);   // 10 — unchanged

// OBJECTS are copied by REFERENCE
let obj1 = { name: "Ali" };
let obj2 = obj1;    // obj2 POINTS to same memory
obj2.name = "Sara";
console.log(obj1.name); // "Sara" — obj1 also changed!

This is one of the most common bug sources for beginners. Understand it deeply.

null vs undefined — The Famous Confusion

let x;                   // undefined — JS set this automatically
let y = null;            // null — YOU intentionally set "no value"

typeof x;               // "undefined"
typeof y;               // "object"  ← famous JS bug, never fixed

x == y;                 // true  (loose equality — both "empty")
x === y;                // false (strict equality — different types)
05

Variables — var, let, const

Keyword Scope Hoisting Reassign Redeclare Use When
var Function / Global Yes (undefined) ✓ ✓ Avoid — legacy
let Block { } TDZ error ✓ ✗ Mutable values
const Block { } TDZ error ✗ (binding) ✗ Most variables!

Hoisting Explained

JS moves variable and function declarations to the top of their scope during the compilation phase — but not their assignments.

console.log(name);  // undefined (not an error!) — var is hoisted
var name = "Ahmed";

console.log(age);   // ReferenceError — let/const in TDZ (Temporal Dead Zone)
let age = 25;

// Functions are FULLY hoisted — you can call them before declaration
greet();              // Works!
function greet() { console.log("Hello"); }

Rule of thumb: Always prefer const by default. Use let only when you know the value will change. Never use var in modern code.

06

Type Coercion & Dynamic Typing

Implicit Coercion — JS tries to "help"

"5" + 2         // "52"  — number converts to string
"5" - 2         // 3    — string converts to number
true + true    // 2    — booleans convert to 1/0
false + "hi"   // "falsehi"
0 == ""         // true  — both coerce to falsy
0 === ""        // false — strict, no coercion

Truthy & Falsy Values

In a boolean context, JS converts any value to true/false:

Falsy (6 values only)

  • false
  • 0 and -0 and 0n
  • "" (empty string)
  • null
  • undefined
  • NaN

Truthy (everything else)

  • "0" (non-empty string)
  • [] (empty array)
  • {} (empty object)
  • -1
  • Infinity
  • function(){}

Always use === (strict equality) in real code. It checks both value and type without coercion. == causes subtle bugs.

07

The Call Stack & Event Loop

The Call Stack

JS uses a stack data structure to track function calls. When a function is called, it's pushed on. When it returns, it's popped off. If the stack is never popped (infinite recursion), you get a Stack Overflow.

function c() { console.log("C"); }
function b() { c(); }
function a() { b(); }
a();

// Call Stack trace:
// [a]
// [a, b]
// [a, b, c]
// [a, b]  ← c returned
// [a]     ← b returned
// []      ← a returned

The Event Loop — How Async Works

JS is single-threaded but non-blocking. Here's why:

// Runtime has 4 parts:
Call Stack      — runs synchronous code
Web APIs        — Browser/Node handles async tasks (setTimeout, fetch, fs)
Callback Queue  — completed async callbacks wait here
Event Loop      — watches: if stack is EMPTY, move from queue → stack

console.log("1");                          // runs immediately → stack
setTimeout(() => console.log("2"), 0); // sent to Web API, then queue
console.log("3");                          // runs immediately → stack

// Output: 1, 3, 2  — even with 0ms delay!
08

Synchronous vs Asynchronous

Synchronous

Code runs top-to-bottom, line by line. Each line waits for the previous to complete. Blocking.

const data = readFile("file.txt");
// BLOCKS until file is read
processData(data);

Asynchronous

Long operations are kicked off, and the program continues. When they complete, a callback/promise handles the result. Non-blocking.

readFile("file.txt", (data) => {
  processData(data); // runs later
});
// continues immediately

Three Async Patterns in JS (Evolution)

// 1. Callbacks (old, leads to "callback hell")
getUser(1, (user) => {
  getPosts(user.id, (posts) => {
    getComments(posts[0].id, (comments) => {
      // Pyramid of doom...
    });
  });
});

// 2. Promises (ES6 - flatter)
getUser(1)
  .then(user => getPosts(user.id))
  .then(posts => getComments(posts[0].id))
  .catch(err => console.log(err));

// 3. Async/Await (modern, reads like sync)
async function loadData() {
  const user  = await getUser(1);
  const posts = await getPosts(user.id);
  return posts;
}

In MERN, you will use async/await everywhere — API calls, database queries, file operations.

Scope & Closures

Scope — Where Can You See a Variable?

let global = "I'm everywhere";  // Global scope

function outer() {
  let x = "outer";               // Function scope

  function inner() {
    let y = "inner";             // Block/function scope
    console.log(x);              // ✓ can see outer's x
    console.log(global);          // ✓ can see global
  }
  // console.log(y) — ✗ Error: y is not defined here
}

Closures — A Function That Remembers Its Birthplace

A closure is when an inner function retains access to its outer function's variables even after the outer function has returned. This is fundamental to React hooks, middleware, and event handlers.

function makeCounter() {
  let count = 0;                // This variable is "closed over"
  return function() {
    count++;
    return count;
  };
}

const counter = makeCounter();
counter();   // 1
counter();   // 2
counter();   // 3 — count persists in closure!

Prototype & Inheritance Model

Prototype Chain

Every JS object has an internal [[Prototype]] link. When you access a property, JS looks up the chain until it finds it or reaches null.

const arr = [1, 2, 3];
arr.map(x => x * 2);   // .map() is NOT on arr itself
// arr → Array.prototype (has .map) → Object.prototype → null

ES6 Class — Syntactic Sugar

class Animal {
  constructor(name) {
    this.name = name;
  }
  speak() {
    return `${this.name} makes a sound`;
  }
}

class Dog extends Animal {
  speak() {
    return `${this.name} barks`;
  }
}

const d = new Dog("Rex");
d.speak();   // "Rex barks"

Under the hood, this is still prototype chains. class is just cleaner syntax introduced in ES6 (2015).

Why JavaScript for MERN?

One Language, Full Stack

Before Node.js, backend was Java/PHP/Python and frontend was JS — two completely different worlds. MERN unifies everything:

// Frontend (React) — same JS
const user = { name: "Ahmed", age: 30 };

// Backend (Node/Express) — same JS
app.get("/user", (req, res) => {
  res.json({ name: "Ahmed", age: 30 });
});

// Database queries (Mongoose) — same JS
const user = await User.findById(id);

JSON is Native

MongoDB stores BSON (binary JSON). APIs send JSON. React uses JSON-like JSX. JavaScript's object literals ARE JSON. Perfect fit — no translation needed.

NPM Ecosystem

2+ million packages. Whether you need authentication (Passport), validation (Joi), email (Nodemailer), or payments (Stripe) — it's one npm install away.
